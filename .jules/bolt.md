## 2026-04-05 - Direct Array Indexing over Sequence Unpacking (Revised)
**Learning:** In tight numerical loops like `equations_of_motion`, duck-typed sequence unpacking (`try...except` combined with element-by-element `type()` checks) introduces constant overhead on every call. However, applying direct index access combined with explicit `float()` casting for *all* input types actually slows down execution when the input is natively a pure Python list or tuple. To optimize for both arrays and native lists, an explicit type check (`if type(state) is tuple or type(state) is list`) combined with simple direct assignment for the native types is the fastest approach, avoiding unnecessary explicit `float()` casts while still providing a fast path for NumPy arrays via the fallback `float()` casting.
**Action:** Use direct indexing combined with an explicit type check for native collections (`tuple` or `list`). Assign elements natively on the fast path, and use explicit `float()` casting on the slow path to properly handle array elements and ensure fast scalar C-level operations.

## 2026-04-09 - Flat Tuples over Nested Lists for Matrix Solvers
**Learning:** In highly repetitive custom numerical solver loops (like `TrimSolver`'s Newton-Raphson `solve_3x3` and `jacobian`), allocating, returning, and accessing nested lists (e.g., `[[a, b, c], [d, e, f]]`) adds measurable memory allocation overhead and repeated index lookup overhead. Returning and accepting flat tuples (e.g., `(a, b, c, d, e, f)`) avoids this instantiation penalty completely and enables extremely fast flat unpacking (`a, b, c, d, e, f = A`) in hot paths.
**Action:** Use flat tuples and flat index-unpacking instead of nested lists for small fixed-size data structures (like 3x3 matrices) inside tight numerical loops.

## 2026-04-10 - Flat Tuples over Lists for High-Frequency Return Types
**Learning:** In hot numerical paths (like `equations_of_motion` called 100,000+ times during integration or Jacobian calculation), returning `list` objects instead of scalar elements introduces significant object allocation overhead. However, unpacking into multiple variables isn't always feasible across boundaries. Returning flat `tuple` objects (e.g., `(udot, wdot...)`) instead of lists completely bypasses dynamic memory allocation overhead in Python. This structural shift yielded a ~10-15% performance improvement in the core ODE loops.
**Action:** When passing fixed-length collections of floats across boundaries in high-frequency numerical loops, return flat tuples instead of lists or arrays to eliminate memory allocation penalties.

## 2026-04-14 - Passing Flat Tuples vs Mutating Lists in Solvers
**Learning:** In the `TrimSolver`'s hot loops (e.g. `objective` and `jacobian` functions evaluated thousands of times), previously an optimization was added to pre-allocate mutable lists (`state = [0.0]*12`) outside the loop and mutate their elements during evaluation. However, profiling revealed that directly creating and passing a flat tuple (e.g. `(u, 0.0, w, 0.0, ... )`) is faster and avoids memory-mutability overhead compared to mutating elements of a list in Python. Unpacking the argument list (`x = [alpha, elevator, throttle]`) into flat variables (`alpha, elevator, throttle`) to pass directly into `objective` and `jacobian` also eliminated list indexing overhead, yielding an additional ~25-30% reduction in `TrimSolver` runtime.
**Action:** Use flat tuple instantiation over pre-allocated list mutation when passing fixed-size data into hot function loops, and pass multiple parameters as flat function arguments instead of wrapped in lists.

## 2026-04-16 - Parameterizing Constants in Outer Solvers
**Learning:** In iterative numerical solvers (like `TrimSolver`), environment variables (e.g., atmospheric density `rho` calculated from `altitude`) may remain completely constant during the entire root-finding loop, yet the core physics functions (`equations_of_motion`) recalculate them natively. By allowing `longitudinal_equations_of_motion` to accept pre-calculated optional constants (`rho=None`), we can bypass redundant, moderately expensive exponentiation operations (~30,000 times per solve) yielding a ~20% performance improvement.
**Action:** When working with iterative solvers, explicitly check if any inputs passed to the inner objective function are completely static during the solver's execution, and hoist their dependent calculations (like `atmosphere_scalar`) outside the loop, passing the results down as optional overrides to the hot core physics calculations.

## 2026-04-16 - Algebraic Trig for Alpha (Revised)
**Learning:** In high-frequency physics functions (`equations_of_motion`), calculating aerodynamic angles like `alpha` requires an `atan2(w, u)` call, and then its sine and cosine are required later (`sin(alpha)`, `cos(alpha)`). This results in 3 expensive trigonometric calls. Since `sin(alpha) = w/V_lon` and `cos(alpha) = u/V_lon` algebraically (where `V_lon = sqrt(u^2 + w^2)`), substituting the two trigonometric calls with simple multiplications provides a ~9-10% performance improvement in the hot path. Note that we must calculate `V_lon` correctly for 3D flight (where `v != 0`, `V` is not simply `sqrt(u^2 + w^2)`), and carefully handle the near-zero velocity edge case.
**Action:** Use algebraic equivalents for trigonometric functions where the sides of the triangle can be safely calculated without trig functions, specifically calculating the appropriate magnitude (`V_lon`) and substituting `math.sin` and `math.cos` calls.

## 2024-05-18 - Flat Tuples for Matrix Columns
**Learning:** In high-frequency numerical loops like `Linearizer.compute_jacobian`, dynamically allocating Python lists for columns (e.g., `A_cols[i] = [...]`) introduces significant memory allocation penalties.
**Action:** Replace dynamically allocated lists with fixed-length flat tuples (e.g., `A_cols[i] = (...)`) in hot paths to eliminate memory allocation overhead and improve execution speed without affecting numpy array conversion.

## 2026-04-24 - Flat Tuple Unpacking in Core Physics Loops
**Learning:** In the `equations_of_motion` function and its longitudinal variant, extracting elements from tuple/list state vectors was previously done using individual index assignments (e.g., `u = state[0]`, `v = state[1]`). Benchmarking revealed that replacing this block with Python's native flat tuple sequence unpacking (e.g., `u, v, w, p, q, r, phi, theta, psi, _, _, z = state`) yields a significant performance improvement (nearly cutting the base Python execution time of the function in half). This is because Python handles unpacking natively at the C-level, eliminating the overhead of repeated list index lookups and assignment operations in tight loops.
**Action:** When extracting multiple contiguous elements from flat sequences in hot numerical paths, use native sequence unpacking instead of individual index lookups.

## 2026-04-28 - Inverse Square Root via math.sqrt()
**Learning:** In Python, calculating the inverse square root using `1.0 / math.sqrt(x)` is measurably faster than using fractional exponentiation (`x ** -0.5`). While `** -0.5` seems syntactically cleaner and mathematically direct, fractional exponentiation relies on the underlying C `pow()` function, which incurs significantly more overhead than a direct `math.sqrt()` call followed by a single division. In hot numerical loops, this can add up. Additionally, assigning variables that are never used later adds unnecessary memory/assignment overhead.
**Action:** When computing an inverse square root, use `1.0 / math.sqrt(x)` (or a local alias like `1.0 / _sqrt(x)`) instead of `x ** -0.5`. Additionally, carefully analyze local variables to ensure intermediate calculation steps aren't keeping variables alive that could be cleanly eliminated.
